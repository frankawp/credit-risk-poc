# 变量挖掘方法论

本文档描述信贷风险变量挖掘的核心方法论和工作流程。

## 核心理念

### 假设驱动

每个变量都应该有明确的业务假设支撑：
- **假设**：为什么这个变量能识别风险？
- **验证**：假设是否成立？效果如何？
- **迭代**：根据验证结果调整假设

### 数据优先

在设计变量之前，先理解数据：
- 数据结构：有哪些表？关联关系是什么？
- 数据质量：缺失率、异常值、分布特征
- 业务含义：字段代表什么？时间窗口如何？

## 工作流程

```
探索 → 设计 → 实现 → 验证 → 迭代
```

### 第一阶段：数据理解

**目标**：建立对数据的全面认知。

**关键问题**：
1. 有哪些数据表？每张表的业务含义是什么？
2. 表之间的关联关系是什么？
3. 目标变量是什么？如何定义？
4. 时间窗口是什么？样本范围如何？

**产出**：
- 数据字典
- 实体关系图
- 数据质量报告

### 第二阶段：假设设计

**目标**：提出有业务意义的变量假设。

**变量分类**：
| 类型 | 描述 | 示例 |
|------|------|------|
| 一致性 | 身份/资料的一致性程度 | 收入稳定性、地址一致性 |
| 高频性 | 短期内的活动频率 | 申请次数、查询次数 |
| 套现倾向 | 套现相关的行为特征 | ATM取现比例、首期还款行为 |
| 历史表现 | 过往信用表现 | 逾期次数、还款率 |

**假设模板**：
```
变量名: velocity_apply_count_30d
业务假设: 短期高频申请的用户风险更高
预期方向: 越高风险越大
计算逻辑: 统计最近30天的申请次数
```

### 第三阶段：代码实现

**目标**：将假设转化为可计算的变量。

**实现原则**：
- 使用标准的数据处理库（pandas, numpy）
- 处理缺失值和异常值
- 添加必要的注释和文档

**代码模板**：
```python
def build_velocity_features(data_frames: dict) -> pd.DataFrame:
    """
    构建 velocity 主题特征。

    变量列表:
    - velocity_apply_count_30d: 最近30天申请次数
    - velocity_apply_count_7d: 最近7天申请次数
    """
    applications = data_frames["applications"]

    # 计算逻辑
    features = applications.groupby("SK_ID_CURR").agg({
        "apply_date": ["count", "nunique"],
    })

    return features
```

### 第四阶段：效果验证

**目标**：验证假设是否成立。

**评估指标**：
- **区分能力**：ROC-AUC、PR-AUC、KS
- **稳定性**：PSI（群体稳定性指标）
- **可解释性**：业务逻辑是否清晰

**评估标准**：
| 指标 | 好 | 一般 | 差 |
|------|-----|------|-----|
| ROC-AUC | >0.65 | 0.55-0.65 | <0.55 |
| Lift@10% | >1.5 | 1.2-1.5 | <1.2 |

### 第五阶段：迭代优化

**目标**：从验证中学习，持续改进。

**迭代方向**：
- 调整变量计算逻辑
- 添加衍生变量
- 组合多个变量
- 放弃无效方向

## 最佳实践

### 变量命名

- 格式：`{theme}_{具体含义}`
- 示例：`velocity_apply_count_30d`
- 避免：缩写、拼音、无意义数字

### 变量文档

每个变量应记录：
1. 变量名称
2. 业务假设
3. 计算逻辑
4. 预期方向
5. 验证结果

### 版本管理

- 使用变量注册表跟踪变量状态
- 记录变量的迭代历史
- 归档废弃的变量

## 常见陷阱

1. **过度拟合**：变量太复杂，在新数据上表现差
2. **数据泄漏**：使用了未来信息
3. **假设模糊**：没有明确的业务假设
4. **忽略缺失值**：缺失值本身可能携带信息
5. **忽视稳定性**：只在训练集有效，测试集失效
