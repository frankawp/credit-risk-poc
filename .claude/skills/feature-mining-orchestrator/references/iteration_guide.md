# 迭代验证指南

如何在变量挖掘过程中进行有效的效果验证和迭代优化。

## 验证流程

### 1. 单变量评估

每个变量单独评估预测能力。

**关键指标**：

| 指标 | 计算方式 | 解读 |
|------|----------|------|
| ROC-AUC | ROC 曲线下面积 | 0.5 无区分力，> 0.6 有效，> 0.7 较好 |
| PR-AUC | PR 曲线下面积 | 对不平衡数据更敏感 |
| 缺失率 | 缺失值占比 | > 50% 通常不可用 |
| 唯一值数 | 非重复值数量 | 1 或 2 可能无区分力 |

**评估标准**：

```
有效变量：
- ROC-AUC >= 0.52 或 PR-AUC >= baseline * 1.02
- 缺失率 < 50%
- 唯一值数 > 2

优秀变量：
- ROC-AUC >= 0.58
- 缺失率 < 10%
```

### 2. 假设验证

对照设计时的假设，验证是否成立。

**验证模板**：

| 原始假设 | 验证结果 | 结论 |
|----------|----------|------|
| 7 天申请次数越高风险越高 | ROC-AUC = 0.56，单调性符合 | 假设成立，保留 |
| ATM 取现比例高风险更高 | ROC-AUC = 0.51，覆盖仅 5% | 假设弱成立，覆盖不足 |
| 首期逾期天数 > 30 天风险高 | ROC-AUC = 0.62，但方向相反 | 假设失败，需调整 |

### 3. 相关性检查

避免保留高度相关的变量。

**处理策略**：

```
相关性 > 0.95：
- 保留 AUC 更高的
- 或保留业务含义更清晰的

相关性 0.8 - 0.95：
- 考虑是否为同一信号的不同表达
- 可保留不同业务视角的
```

## 迭代优化

### 场景一：假设成立，效果一般

**原因分析**：
- 变量计算逻辑不够精准
- 时间窗口选择不当
- 缺失值处理不当

**优化方向**：
- 调整窗口大小（如 7d → 3d）
- 改变聚合方式（如 sum → mean）
- 增加条件筛选

### 场景二：假设不成立

**原因分析**：
- 业务理解有误
- 数据质量问题
- 样本偏差

**优化方向**：
- 反向思考：是否方向相反？
- 检查数据分布：是否有异常值干扰？
- 检查样本：是否某类客户占比过高？

### 场景三：覆盖不足

**原因分析**：
- 数据表关联缺失
- 条件过于严格
- 历史数据不足

**优化方向**：
- 放宽条件
- 使用默认值填充
- 寻找替代数据源

## 记录模板

每次迭代记录以下内容：

```markdown
## 变量迭代记录

### 变量名：velocity_prev_count_7d

### 第一版
- 假设：7 天内申请次数越高风险越高
- 计算：count(DAYS_DECISION >= -7)
- 结果：ROC-AUC = 0.54，覆盖 80%
- 结论：有效但不够强

### 第二版（迭代）
- 调整：改为 3 天窗口
- 计算：count(DAYS_DECISION >= -3)
- 结果：ROC-AUC = 0.58，覆盖 60%
- 结论：效果提升，覆盖下降，保留两个版本

### 最终状态
- 保留：velocity_prev_count_7d, velocity_prev_count_3d
- 理由：不同时间窗口有不同业务含义
```

## AI 协作要点

在迭代过程中，AI 会：

1. **自动评估**：对每个变量计算标准指标
2. **对比假设**：验证方向是否符合预期
3. **推荐调整**：基于结果建议优化方向
4. **记录过程**：自动生成迭代日志

你需要：

1. **提供业务判断**：AI 无法替代业务直觉
2. **确认方向**：对关键决策做确认
3. **补充背景**：解释异常结果的业务原因
4. **决定取舍**：最终保留哪些变量

## 常见问题

### Q: AUC 很低但业务直觉认为应该有效？

可能原因：
- 数据质量问题（异常值、缺失）
- 样本偏差
- 变量定义与业务直觉有偏差

建议：
- 先检查数据分布
- 尝试不同口径
- 与业务方确认逻辑

### Q: 两个变量高度相关，保留哪个？

考虑因素：
- 业务含义：哪个更易解释
- 计算成本：哪个更简单
- 覆盖率：哪个覆盖更广
- 稳定性：哪个更稳定

### Q: 什么时候停止迭代？

满足以下条件时可停止：
- 新变量不再显著提升效果
- 迭代成本 > 潜在收益
- 达到业务方接受的标准

记住：**没有完美的变量，只有够用的变量**。
