# 信贷客户变量挖掘方法

## 1. 方法目标

本方法用于把公开数据或业务数据中的客户信息、申请行为、征信行为和贷后行为，系统化地转化为一组可复用的客户级变量。

目标不是“多造特征”，而是建立一套稳定方法来回答三个问题：

1. 哪些变量能更好地描述客户风险或可疑行为
2. 哪些变量具备业务解释，可以沉淀为规则或画像
3. 哪些变量在模型中确实带来增益，哪些只是噪音或重复信号

## 2. 适用场景

这套方法适用于：

- 信贷风控变量挖掘
- 反欺诈变量挖掘
- 客户画像与风险标签体系建设
- 公开比赛数据的 POC 验证

## 3. 核心原则

### 3.1 先业务分层，再算特征

变量挖掘不要从字段开始，而要从业务问题开始。建议先把变量池划成几类：

- 身份一致性
- 申请密度与行为频率
- 征信外部暴露
- 贷后恶化与失联风险
- 群体风险和弱关系特征

### 3.2 先客户级聚合，再模型化

原始行为表通常是明细表，直接建模容易失控。应先统一到客户级：

- 主键优先统一为 `SK_ID_CURR`
- 每个变量都必须能说明“聚合口径是什么”
- 宽表是建模输入，不是原始事实表替代

### 3.3 先可解释，再扩展复杂度

第一阶段优先做：

- 口径清晰
- 业务可解释
- 易验证
- 易回看样本

的变量。复杂图特征、embedding 特征或高阶自动组合，应放到后续阶段。

### 3.4 行为信号优先于静态信号

静态身份信息很重要，但真正稳定的高风险信号往往来自行为轨迹：

- 短期申请密集
- 近期征信激增
- 贷后快速恶化
- 资金使用方式异常

因此变量设计时应优先行为类，再补静态一致性类。

## 4. 标准工作流

### Step 1. 数据理解

先回答这几件事：

- 主表是谁
- 客户主键是什么
- 事件表有哪些
- 时间字段是绝对时间还是相对时间
- 标签是否是真实业务标签还是弱监督标签

输出：

- 数据总览
- 表关系说明
- 数据质量注意事项

### Step 2. 变量母题拆解

围绕业务目标拆出变量母题，不要一开始就列字段名。

以反欺诈为例，可拆为：

1. 信息不一致性
2. 短期高频行为
3. 套现与团伙特征
4. 群体高风险画像
5. 历史拒绝与申请链路异常

输出：

- 变量主题树
- 每个主题下的候选母题

### Step 3. 字段到变量映射

把每个母题映射到具体字段与表：

- 哪张表能支持这个母题
- 计算是计数、比率、差值、窗口、排序还是分组编码
- 聚合主键是否统一

输出：

- 表与变量映射表
- 字段依赖清单

### Step 4. 变量原型构造

每个变量先做原型版本，不追求完美，先验证是否有信号。

变量原型常见形式：

- Count
- Ratio
- Time Gap
- Change Rate
- Window Count
- Status Transition
- Group Risk Encoding

输出：

- 原型特征宽表
- 特征元数据

### Step 5. EDA 与样本回看

变量不是算出来就结束，而要做两层验证：

1. 分布验证
   - 是否极端稀疏
   - 是否异常爆炸
   - 缺失是否可解释
2. 样本验证
   - 命中的客户是否真的更异常
   - top 样本是否符合业务直觉

输出：

- 分布摘要
- 样本回看清单

### Step 6. 特征筛选

变量挖掘完成后，必须做筛选，不能把所有变量全塞给模型。

筛选维度至少包括：

- 基础质量
  - 缺失率
  - 常数 / 近常数
  - 极端值
- 单变量有效性
  - ROC-AUC
  - PR-AUC
  - Recall@TopK
  - Lift
- 相关性去重
  - 高相关组保留代表特征
- 多变量贡献
  - 在全模型中的重要性

输出：

- scorecard
- correlation groups
- selected features

### Step 7. 增量验证

最终必须回答：

- 新变量相比 baseline 提升了多少
- 哪些主题真正有贡献
- 哪些变量只是重复信息

输出：

- 全量 vs 筛选后对比
- 主题级贡献判断
- 下一阶段迭代方向

## 5. 变量设计模板

每个变量都建议按统一模板描述：

- `feature_name`
- `source_table`
- `grain`
- `formula`
- `window_or_grouping`
- `risk_direction`
- `business_meaning`
- `known_issues`

这样做的好处是：

- 后续回看时不需要重新读代码
- 可以直接做变量手册
- 有利于规则引擎和模型共用变量定义

## 6. 变量主题的推荐优先级

在信贷客户变量挖掘中，建议优先级如下：

1. 贷后恶化和还款异常
2. 申请密度与征信短期活跃
3. 群体风险编码
4. 静态身份一致性
5. 渠道、商品、拒绝原因等场景变量
6. 图关系和复杂时序特征

原因是：

- 前三类通常更稳定
- 后三类更依赖业务上下文和真实主键

## 7. 方法的产出物规范

每轮变量挖掘建议固定产出：

- 数据说明文档
- 变量手册
- 宽表文件
- 特征筛选报告
- 模型对比结果
- 分析过程归档

这样变量挖掘才不是一次性实验，而是一个可追溯的方法体系。

## 8. 这套方法在当前项目中的体现

当前项目已经按这套方法落地了第一阶段：

- 数据理解：`docs/data_knowledge/`
- 变量手册：`docs/feature_catalog.md`
- 分析归档：`docs/analysis_runs/`
- 特征筛选：`feature_scorecard.csv`、`correlation_groups.csv`
- 模型验证：`baseline_metrics_all.json`、`baseline_metrics_selected.json`

因此，后续新增变量时，应继续遵守这套方法，而不是临时拼接脚本。
